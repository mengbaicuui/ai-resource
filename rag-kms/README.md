## 🔮 RAG 核心概念

### 什么是 RAG？

**RAG = Retrieval-Augmented Generation (检索增强生成)**

```
用户查询 → 检索 → 增强 → 生成
              ↓
         私有知识库
```

**核心流程：**
1. 从私有知识库中检索相关信息
2. 将检索结果作为上下文注入 prompt
3. LLM 基于增强后的 context 生成答案

### 阶段一：索引构建 (Indexing)

#### 1. 文档分块 (Chunking) 策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **固定长度** | 按字数/字符数分块 | 通用场景 |
| **句子分割** | 按句子边界分块 | 对话系统 |
| **段落分割** | 按自然段落分块 | 文档问答 |
| **递归分块** | 按层级结构递归分割 | 结构化文档 |
| **语义分块** | 基于语义相似度分块 | 复杂文档 |

#### 2. Embedding 模型选择

| 模型 | 说明 | 特点 |
|------|------|------|
| **OpenAI (text-embedding-ada-002)** | OpenAI 官方 | 通用效果好 |
| **Cohere** | Cohere Embed | 多语言支持 |
| **BGE** | 国产开源 | 中文优化 |
| **Jina Embed** | Jina AI | 长文本支持 |
| **Local (all-MiniLM)** | 本地部署 | 隐私优先 |

#### 3. 向量数据库

| 数据库 | 类型 | 特点 |
|--------|------|------|
| **Pinecone** | 云托管 | 全托管、高可用 |
| **Weaviate** | 开源+云 | 混合检索强 |
| **Milvus** | 开源 | 超大规模场景 |
| **Qdrant** | 开源 | 高性能向量检索 |
| **Chroma** | 开源 | 轻量级、本地开发 |
| **FAISS** | 开源 | Facebook出品、内存密集型 |

### 阶段二：检索增强 (Retrieval)

#### 检索策略对比

| 策略 | 说明 | 优缺点 |
|------|------|--------|
| **纯向量检索** | 语义相似度匹配 | 语义理解好，精确词匹配弱 |
| **关键词检索 (BM25)** | 词频统计排序 | 精确匹配强，语义理解弱 |
| **混合检索** | 向量+关键词混合 | 平衡两者，效果最佳 |
| **重排序 (Reranker)** | Cross-Encoder 重排 | 精度更高，延迟增加 |

#### RAG 优化技巧

| 技巧 | 说明 |
|------|------|
| **Query Expansion** | 扩展查询以获得更多相关结果 |
| **HyDE** | 用 LLM 生成假设性答案再检索 |
| **Self-RAG** | 让 LLM 自己判断是否需要检索 |
| **Corrective RAG** | 评估检索结果质量，必要时重检 |
| **Memory RAG** | 保存历史对话上下文 |

---

### 阶段三：生成增强 (Generation)

#### Prompt 模板示例

```
你是一个问答助手。请基于以下参考内容回答用户问题。
如果参考内容中没有相关信息，请如实说明。

【参考内容】
{context}

【用户问题】
{question}

【回答】
```

#### LLM 选择考量

| 考量因素 | 说明 |
|----------|------|
| **上下文长度** | 是否支持足够长的 context |
| **成本** | API 调用成本 |
| **延迟** | 生成速度 |
| **中文能力** | 中文理解和生成质量 |
