## 1. 底层核心技术

Claude Code 的强大在于它不是简单的"对话框+代码补全"，而是一套基于 **Agentic Loop（智能体循环）** 的闭环系统。

---

### A. 渐进式披露 (Progressive Disclosure) 与上下文管理

| 特性 | 说明 |
|------|------|
| **按需加载** | 不一次性读入所有文件，通过 SKILL.md 架构实现"按需加载" |
| **Skill 元数据** | 初始只加载 Skill 的名称和描述，触发时才读取详细指令 |
| **Token 节省** | 大幅降低 Token 损耗，保持长对话的逻辑连贯 |
| **上下文压缩** | 长轮次对话中自动对历史信息摘要压缩，防止触达上限 |
| **超长上下文** | Opus 4.6 已支持 **1M Token** 窗口 |

---

### B. 模型上下文协议 (MCP - Model Context Protocol)

**定位**：Claude Code 的"外交官"

| 特性 | 说明 |
|------|------|
| **开放标准** | Anthropic 发起的开源标准 |
| **跨平台连接** | 允许 Agent 连接外部数据源（Jira, Slack, PostgreSQL, Google Drive） |
| **即插即用** | 传统工具调用是硬编码的，MCP 让 Claude Code 具备生态连接能力 |

**MCP vs 传统 Tool Use：**
- 传统 Tool Use：硬编码绑定
- MCP：标准化协议，类似 USB 接口，设备可插拔

---

### C. 多智能体协作与任务锁定 (Sub-agents & Task Locking)

**架构**：处理复杂任务时，Claude Code 可生成多个 **Sub-agents（子智能体）**

| 特性 | 说明 |
|------|------|
| **并发控制** | 通过 `.claude/tasks/` 文件系统通信 |
| **任务锁定** | 利用 Git 同步机制防止代码冲突 |
| **智能调度** | 如果一个 Agent 正在修改某模块，另一个 Agent 会自动转去处理其他任务 |

---

### D. 安全沙箱与执行引擎 (Secure Sandbox)

| 特性 | 说明 |
|------|------|
| **直接执行** | 可直接运行 Bash 指令和 Python 脚本 |
| **自动化测试** | 用于自动化测试、环境配置、代码验证 |
| **Plan-Action-Verify** | 修改代码后主动运行测试，测试通过才提交 |

## 2. 核心机制：Agentic Loop

```
┌─────────────────────────────────────────────────────────────┐
│                    Agentic Loop                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────┐                                               │
│   │   Plan  │  ← 分析任务，制定执行计划                      │
│   └────┬────┘                                               │
│        ↓                                                    │
│   ┌─────────┐                                               │
│   │  Act   │  ← 执行代码修改/Bash命令/Search               │
│   └────┬────┘                                               │
│        ↓                                                    │
│   ┌─────────┐                                               │
│   │ Verify  │  ← 运行测试，验证结果                          │
│   └────┬────┘                                               │
│        ↓                                                    │
│   ┌─────────┐                                               │
│   │ Commit  │  ← 测试通过才提交                              │
│   └─────────┘                                               │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 3. 与其他主流 Agent 工具的异同

### 横向对比

| 维度 | Claude Code | Cursor (Agent) | GitHub Copilot | Aider | Devin |
|:---|:---|:---|:---|:---|:---|
| **交互形态** | Terminal (CLI) | IDE 原生集成 | 插件/侧边栏 | Terminal (CLI) | Web Dashboard |
| **底层架构** | Client-Server (Anthropic) | VS Code 分支 | VS Code 插件 | 多模型适配器 | 全托管云端 |
| **核心优势** | 深度推理 + Git/终端无缝整合 | 极致 UI + 代码补全体验 | 生态最强，大厂合规首选 | 多模型切换 | 全自动"甩手掌柜" |
| **上下文处理** | 渐进式披露 + 1M Token | RAG + Embeddings | 片段式 RAG | 全仓库扫描 (Repo map) | 长期记忆 + 全量上下文 |
| **模型灵活性** | 仅限 Claude (Sonnet/Opus) | 可切换主流模型 | 仅限 OpenAI/Claude | 极高（支持任何 LLM） | 自研模型为主 |
| **MCP 支持** | ✅ 原生 | ✅ | ❌ | ❌ | ❌ |
| **Sub-agents** | ✅ | ✅ | ❌ | ❌ | ✅ |
| **Plan-Action-Verify** | ✅ | ✅ | ❌ | ❌ | ✅ |

---

### 关键异同点

#### Claude Code vs. Cursor

| 同 | 异 |
|---|---|
| 都具备多文件修改和自主思考能力 | Cursor 像"超级编辑器"，强调视觉 Diff；Claude Code 像"资深架构师"，全链路自动化程度更高 |

#### Claude Code vs. Aider

| 同 | 异 |
|---|---|
| 都是终端重度用户福音 | Aider 优势在开源+多模型；Claude Code 优势在**原厂优化的 Agent Skill 系统**，复杂逻辑推理（尤其 Opus 4.6）在 SWE-bench 测试中通常领先 |

#### Claude Code vs. Devin

| 同 | 异 |
|---|---|
| 都追求高阶自主性 | Devin 是独立工作空间，适合完全外包一个功能；Claude Code 是**本地开发流程的增强**，不接管环境，作为终端插件协助 |

## 6. PM 视角总结

### 核心价值

| 价值点 | 说明 |
|--------|------|
| **结构化思维** | Skill 系统将纯 Prompt 转化为结构化思维模型，具备持久化记忆 |
| **业务赋能** | 通过 MCP 协议，Agent 不只处理文本，能直接读取生产数据库、调取客户反馈记录 |
| **自动化闭环** | Plan-Action-Verify 机制确保代码质量，无需人工介入每一步 |

### 适用场景

- ✅ 本地开发流程增强
- ✅ 复杂代码库理解
- ✅ 自动化测试和部署
- ✅ 多文件重构任务
- ✅ 需要深度推理的架构设计

### 不适用场景

- ⚠️ 需要完全托管的云端工作空间
- ⚠️ 非技术用户（CLI 门槛）
- ⚠️ 需要最强多模型灵活性的场景
